### 2.3 可迁移的产品管理模式

#### 📌 模式 1：角色激活三要素

> **任何新角色的有效运行需要：明确输入 + 清晰边界 + 触发机制**

- **输入**：需求从哪里来？（用户反馈/数据分析/战略规划）
- **边界**：什么是我该做的？什么是别人做的？
- **触发**：什么情况下启动协作？（需求评审/技术方案/风险预警）

**适用场景：** 新 Agent 角色创建、新团队成员 onboarding、跨部门协作建立

---

#### 📌 模式 2：基础设施≠实际运行

> **文档/工具/配置的完成 ≠ 角色的有效运行**

- 基础设施是必要条件，非充分条件
- 需要**首次真实工作循环**来验证机制有效性
- 建议：新角色创建后，强制在 7 天内完成至少 1 次真实工作闭环

**适用场景：** 新流程上线、新工具引入、组织变革

---

#### 📌 模式 3：协作机制需要"最小可行事件"

> **定义一个最小的、具体的协作事件，而非抽象的"建立机制"**

- ❌ 抽象目标："与 Architect 建立沟通机制"
- ✅ 具体事件："完成 REQ-001 需求的技术可行性评估"

**适用场景：** 跨角色协作、跨部门项目、敏捷团队启动

## 三、【变】如何优化需求管理流程？

### 3.1 当前流程问题

```
[需求产生] → ??? → [需求池] → ??? → [与 Architect 评审] → ??? → [排期]
              ↑                      ↑                     ↑
           无输入渠道            无触发规则           无优先级标准
```

### 3.2 优化方案

#### 改进 1：建立需求输入渠道

| 渠道 | 说明 | 负责人 |
|------|------|--------|
| 用户反馈 | 从飞书消息/会话中提取产品需求 | ProductOwner 主动扫描 |
| 战略规划 | 从 strategist 的 Q2 规划中拆解产品需求 | ProductOwner + strategist |
| 技术驱动 | Architect 提出的技术改进需求 | Architect → ProductOwner |
| 数据分析 | 从系统使用数据中发现优化点 | sysadmin → ProductOwner |

#### 改进 2：定义协作触发条件

**必须与 Architect 协商的场景：**
- [ ] 新需求进入 P0/P1 优先级
- [ ] 需求涉及技术架构变更
- [ ] 需求工作量评估（用于排期）
- [ ] 技术债务处理优先级
- [ ] 上线时间承诺

#### 改进 3：需求状态流转规则

```
待评估 → 已评估（完成技术可行性分析）→ 排期中（确定 Sprint）→ 进行中 → 已完成

流转条件：
- 待评估→已评估：ProductOwner 完成业务价值分析 + Architect 完成技术评估
- 已评估→排期中：与老板确认优先级，分配至具体 Sprint
- 排期中→进行中：Sprint 开始
- 进行中→已完成：验收通过
```

## 五、可迁移模式总结

### 📦 模式卡片 1：角色激活三要素

```
模式名称：角色激活三要素
适用场景：新 Agent/新岗位/新流程启动
核心要素：
  1. 明确输入（需求/任务从哪里来）
  2. 清晰边界（职责范围，与他人的区分）
  3. 触发机制（什么情况下启动协作/工作）
检查清单：
  □ 输入渠道已定义且可操作
  □ 边界文档化且相关方已知晓
  □ 至少有 1 个具体触发事件（非抽象目标）
```

### 📦 模式卡片 2：最小可行协作事件

```
模式名称：最小可行协作事件
适用场景：跨角色/跨部门协作启动
核心方法：
  用具体事件替代抽象目标
  ❌ "建立沟通机制"
  ✅ "完成 REQ-001 的技术可行性评估"
关键特征：
  - 有明确输入（具体需求/问题）
  - 有明确输出（评审记录/决策）
  - 有明确参与方（谁发起、谁响应）
  - 有时间边界（何时完成）
```

### 📦 模式卡片 3：基础设施验证闭环

```
模式名称：基础设施验证闭环
适用场景：新系统/新流程/新工具上线
核心原则：文档/工具/配置完成 ≠ 有效运行
验证步骤：
  1. 基础设施搭建（文档、工具、配置）
  2. 定义最小可行工作循环（1 次完整流程）
  3. 执行并记录问题
  4. 优化机制
  5. 进入常规运行
时间要求：7 天内完成首次闭环
```
