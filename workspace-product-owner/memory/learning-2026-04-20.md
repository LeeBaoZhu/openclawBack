## 三、【变】如何优化需求管理流程？

### 3.1 当前流程问题

```
[需求录入] → [评审排期] → [开发] → [上线] → ???
     ↑            ↑          ↑         ↑
     │            │          │         │
  来源单一    协作通道   无中期    无价值
  依赖观察    未验证    检查      验证
```

**问题清单：**
1. **协作通道未验证** — 开始协作前未确认目标 Agent 是否可用
2. **中期检查缺失** — Sprint 1/2 均无中期同步，进度黑盒
3. **需求来源被动** — 依赖用户触发或运维观察，缺乏系统性收集
4. **价值验证未定义** — 新需求 KPI 未定义"上线后如何验证价值"

### 3.2 优化方案

#### 改进 1：协作通道前置验证

**在首次需求评审前，必须完成：**
- [ ] 确认目标 Agent 已注册（sessions_list 或 agents_list 检查）
- [ ] 测试 sessions_send 通信有效（发送测试消息并确认响应）
- [ ] 建立同步频率约定（如：每周一/三/五进度同步）

**执行规则：** 未通过协作通道验证的需求，不进入排期

#### 改进 2：Sprint 中期检查强制化

| Sprint 长度 | 检查点时间 | 检查内容 | 输出 |
|-------------|------------|----------|------|
| ≤5 天 | 第 3 天 | 进度确认 + 风险识别 | 简短消息 |
| 6-10 天 | 第 4 天 + 第 7 天 | 进度 + 演示 + 调整 | 消息 + 截图 |
| >10 天 | 每周固定时间 | 周进度 + 需求澄清 | 消息 + 文档链接 |

**执行规则：** Architect 必须在检查点主动发送 `[Sprint 中期同步]` 消息，ProductOwner 记录到 decision-log.md

#### 改进 3：需求价值验证指标定义

**在需求评审时，必须输出"价值验证卡片"：**

```
需求名称：[填写]
验证时机：上线后 [X] 周
验证指标：
  - 指标 1：[名称]，baseline=[当前值]，目标值=[期望值]
  - 指标 2：[名称]，baseline=[当前值]，目标值=[期望值]
验证方法：[数据统计/用户访谈/A-B 测试]
负责人：[ProductOwner/Architect]
```

**示例（运营中心 P0 页面）：**
- 验证时机：上线后 2 周
- 验证指标：页面配置平均耗时从 2 天降至 4 小时
- 验证方法：追踪 5 个新页面的配置时间
- 负责人：ProductOwner

#### 改进 4：建立 ProductOwner 主动跟进机制

**每周固定工作（不依赖外部触发）：**

| 时间 | 任务 | 输出 |
|------|------|------|
| 周一 10:00 | 每周学习反思（本 Cron） | learning-YYYY-MM-DD.md |
| 周三 10:00 | Sprint 进度检查 | decision-log.md 更新 |
| 周五 15:00 | 需求扫描（会话 + 日志） | 潜在需求清单 |
| 周日 20:00 | 下周计划制定 | 周计划文档 |

### 📦 模式卡片 1：结果导向的 KPI 合并原则

```
模式名称：结果导向的 KPI 合并原则
适用场景：能力建设类项目、工具类产品、从 0 到 1 的功能开发
核心原则：当多个 KPI 存在因果关系时，合并为单一结果导向指标
识别方法：
  - 画出 KPI 之间的因果链
  - 找到最终业务结果（通常是交付/收入/满意度）
  - 将中间能力指标降级为过程指标或验收条件
风险控制：为弹性目标准备备选方案，确保底线目标不受影响
```

```
模式名称：Agent 协作健康度检查清单
适用场景：多 Agent 协作、跨团队项目、依赖外部交付的场景
检查频率：每周至少 1 次
检查项：
  □ 目标 Agent 是否已注册（sessions_list 检查）
  □ 最近一次同步时间（超过 3 天需预警）
  □ 同步内容是否有实质性进展
  □ 需求状态是否与计划一致（进度偏差 >20% 需升级）

```
