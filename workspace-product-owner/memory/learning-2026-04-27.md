### 2.3 可迁移的产品管理模式

#### 📌 模式 1：故障驱动的需求发现（Fault-Driven Discovery）

> **运维故障是高质量需求的富矿——每次故障都暴露一个"未被产品化的痛点"**

- **识别方法：** 故障复盘时问"这个问题能否被预防？如何产品化？"
- **转化路径：** 故障现象 → 根因分析 → 预防机制 → 产品需求
- **案例：** Git Token 过期 5 天才发现 → REQ-003 自动化凭证健康检查
- **类似场景：** 
  - 日志文件膨胀 → 日志轮转自动化
  - Cron 任务失败 → 任务监控看板
  - API 限流触发 → 速率管理工具

**适用场景：** 运维系统、内部工具、基础设施产品

**关键问题清单：**
1. 这个故障是否可以被提前预警？
2. 预警的触发条件是什么？
3. 预警后谁来处理？处理流程是什么？
4. 能否自动化处理，减少人工介入？

---

#### 📌 模式 2：延期决策的文档化强制规则

> **任何上线日期变更都必须在决策日志中记录，无论变更原因**

- **规则：** 延期 >3 天 = 必须记录（原因 + 新计划 + 影响评估）
- **记录模板：**
  ```
  需求 ID：[REQ-XXX]
  原计划上线：[日期]
  新计划上线：[日期]
  延期天数：[X 天]
  延期原因：[技术阻塞/需求变更/资源不足/其他]
  影响评估：[对业务/对其他需求/对用户的影响]
  决策人：[谁批准的延期]
  记录时间：[YYYY-MM-DD]
  ```
- **执行机制：** Architect 延期时主动通知 ProductOwner → ProductOwner 记录到 decision-log.md

**适用场景：** 任何有明确上线日期的项目

**本周教训：** REQ-002 延期 14 天（04-10 → 04-24），决策日志中无任何记录，导致 ProductOwner 在 04-23 心跳检查时才发现。

---

#### 📌 模式 3：自动化系统的容错设计原则

> **自动化任务必须考虑"触发环境变化"的容错能力**

- **脆弱点：** Cron 任务依赖 Gateway 进程持续运行，Gateway 重启 = 错过触发窗口
- **防御策略：**
  - **补偿机制：** 任务未执行时自动补推（如 AI 日报 04-27 的处理）
  - **健康检查：** 定期检查自动化任务是否按预期执行
  - **降级方案：** 核心自动化任务失败时，有手动执行的备选路径
- **设计原则：** "自动化 = 自动执行 + 自动补偿 + 自动告警"

**适用场景：** Cron 任务、定时推送、自动化巡检、CI/CD 流水线

**本周教训：** AI 日报因 Gateway 重启错过 7:30 触发窗口，凌晨 10:00 才发现——说明"自动化任务失败检测"本身也需要自动化。

| Gateway 进程是否健康 | 每小时 | 进程异常 | 自动重启 + 告警 |

**价值：** 将"事后发现"转为"事前预防"，与 REQ-003 凭证健康检查形成"基础设施双监控"
