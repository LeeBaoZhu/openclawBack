## 2. 【想】这些问题背后的模式是什么？

### 模式一：配置变更的单点脆弱性 🔴🔴（核心发现）

**事实链**：用户在 `agents.defaults.compaction` 添加 `notifyUser: true`（当前版本不支持）→ hot-reload 被跳过 → feishu channel 被**反注册** → 全部 13 个 cron 任务 00:08 起无法投递 → 用户 08:10 主动询问「为啥两天没收到通知」

**深层模式**：
- **hot-reload 失败 ≠ 配置不生效**，而是**现有功能被破坏**（channel 插件被反注册）
- `agents.defaults` 级别的配置变更影响全局，任何错误都是全局性故障
- 不支持字段不是「被忽略」，而是触发灾难性副作用
- **静默故障**：Gateway 进程表面运行正常，cron timer 继续 re-arm，但没有 channel 能输出消息 — 用户不主动问，系统不会自己报警

**与历史教训的呼应**：
- 3/26 例：「delivery 缺 channel 字段」→ cron 报错
- 4/3 例：「exec 审批双层配置」→ agent 执行失败
- **本次升级**：从不支持字段 → **全局 channel 反注册** → 静默故障 → 人工发现

### 模式二：凭证生命周期存在断点

5/7 的 284 次 HTTP 401（持续 9 小时）自愈，但根因未明。可能原因：
- 飞书 access token 过期后自动续期存在延迟窗口
- Gateway 重启后凭证缓存未正确重建
- 与 3/25「飞书 system busy 260+ 次连续重试」是同类问题：**token 刷新期间的临时不可用**无人感知，直到统计异常才发现

### 模式三：噪音淹没信号

日均 90% 的 Error/Warn 是已知重复类型（persona.md 缺失、EISDIR、ENOENT、symlink skip）。这些噪音让真正的信号（401 波、channel 反注册）在日报中显得「与其他 error 差不多」，降低了人的警觉性。

### 模式四：Gateway 高峰期重启集群

## 3. 【变】如何优化维护流程？

### 变更 1：建立「配置变更安全三步法」

**Before**：改配置 → hot-reload 自动生效（或跳过，无人知）
**After**：
1. **改前验证**：确认字段在当前版本支持（查 schema / `openclaw config show` 对照）
2. **改后验证**：发送测试消息确认 channel 仍可用
3. **失败告警**：hot-reload 失败应立即通知，而非静默

### 变更 2：引入「信号/噪音」分级过滤

当前日报把所有 Error 并列展示，掩盖了真正重要的问题。建议：
- **L1 信号**：新增类型、影响功能的错误（如 401 波、channel 反注册）
- **L2 噪音**：已知重复类型（persona.md、EISDIR），仅统计趋势，不逐条列出
- **L3 忽略**：预期行为（如 Agent 按需读未创建文件）

### 变更 3：增加 post-config-change 自动化检测

在每次 `openclaw.json` 修改后，增加一步：
```bash
# 检查 hot-reload 是否成功
grep "config reload" /tmp/openclaw.log | tail -1
# 验证 channel 是否在线
# (通过 cron run 或直接发测试消息)
```

## 可迁移维护模式（本周提炼 3 个）

### 模式 1：「Hot-Reload 级联故障」模式

**症状**：配置修改后，看似无关的功能突然全部失效
**根因**：不支持字段 → hot-reload 中断 → 插件反注册 → 全局功能退化
**适用场景**：任何涉及 `agents.defaults` 或 `plugins` 级别的配置修改
**防御**：改后验证 channel 可用性；`agents.defaults` 级别的修改视为高危操作

### 模式 2：「静默故障发现」模式

**症状**：系统表面运行正常（Gateway 进程 alive、cron timer re-armed），但实际输出管道全断
**发现途径**：用户主动询问「为什么没收到消息」
**根因**：输出通道（channel）的状态不在 cron 健康检查范围内
**防御**：周期性发送「心跳测试消息」验证端到端投递；cron delivery failure 应有独立告警

### 模式 3：「噪音归一化」模式

**症状**：日报充斥着大量已知重复 Error/Warn，真正新增的问题被淹没
**根因**：只做分类统计，不做「新增类型」标记，所有 Error 视觉权重相同
**改进**：日报用「新增类型」作为第一维度，已知类型归入趋势统计块；用 🔴 和数字标记真正需要关注的新问题
