## 2. 【想】这些问题背后的模式是什么？

### 2.1 配置类问题（占比 60%）

| 问题 | 模式归类 | 根本原因 |
|------|----------|----------|
| lifehelper open_id 错误 | **跨应用配置混淆** | 飞书多应用架构下，同一用户在不同应用有不同 ID |
| exec 审批双层配置 | **配置分层不同步** | OpenClaw 有请求策略 + 执行策略两层，只改了一层 |
| Feishu 联系人权限缺失 | **应用权限未开通** | 新建应用后未在开放平台手动开通权限 |

**模式总结**: 
> **配置分散 + 隐式依赖 = 故障隐患**
> 
> 当配置信息分散在多处（openclaw.json + exec-approvals.json）或依赖隐式知识（飞书 open_id 跨应用不同）时，极易出错。

### 2.2 定时任务类问题（占比 30%）

| 问题 | 模式归类 | 根本原因 |
|------|----------|----------|
| lifehelper Cron 连续失败 | **投递配置不完整** | delivery 缺少 channel 字段，多渠道环境下无法自动推断 |
| Cron 任务错误未及时发现 | **巡检机制不完善** | 依赖日报被动发现，缺少主动监控 consecutiveErrors |

**模式总结**:
> **定时任务 = 配置正确性 × 持续监控**
> 
> 任务创建时配置不完整会导致立即失败；缺少监控会导致失败持续累积。

### 2.3 系统稳定性问题（占比 10%）

| 问题 | 模式归类 | 根本原因 |
|------|----------|----------|
| Gateway 关闭超时 | **进程清理异常** | 重启时资源未正常释放（偶发） |
| Feishu API 502 | **外部服务波动** | 飞书服务端临时故障（transient） |

**模式总结**:
> **外部依赖波动不可控，但可快速恢复**
> 
> 此类问题通常自动恢复，重点是区分 transient 错误和系统性故障。

## 3. 【变】如何优化维护流程？

### 3.1 配置管理优化

**当前问题**:
- 配置分散在多处（openclaw.json、exec-approvals.json、jobs.json）
- 隐式依赖未文档化（如飞书 open_id 跨应用不同）
- 配置变更无版本追踪

**改进方向**:
1. **建立配置清单** — 列出所有关键配置项及其位置，修改时逐项核对
2. **固化隐式知识** — 将「飞书 open_id 跨应用不同」等知识写入 MEMORY.md 和配置规范
3. **配置变更日志** — 每次修改记录时间、原因、变更内容、影响范围

### 3.2 定时任务监控优化

**当前问题**:
- 依赖日报被动发现 Cron 失败
- consecutiveErrors 达到 4 次才引起注意
- 缺少失败告警机制

**改进方向**:
1. **Cron 状态主动检查** — 每日巡检时显式检查 jobs.json 的 lastRunStatus 和 consecutiveErrors
2. **失败阈值告警** — consecutiveErrors ≥ 2 时立即报告，不要等到 4 次
3. **测试任务验证** — 修改配置后创建一次性测试任务验证投递成功

### 3.3 问题诊断流程优化

**当前问题**:
- exec 审批问题花费较长时间定位（双层配置）
- 缺少系统性诊断清单

**改进方向**:
1. **建立诊断清单** — 针对常见问题（Cron 失败、权限错误、exec 报错）建立排查步骤
2. **优先使用 CLI 命令** — 如 `openclaw approvals get` 直接查看有效策略，避免猜测
3. **记录诊断路径** — 将排查过程写入 memory 文件，便于下次快速定位

## 📦 可迁移的维护模式

### 模式 1: 配置分层同步原则
> **适用场景**: 任何存在多层配置的系统（如 OpenClaw 的 openclaw.json + exec-approvals.json）
> 
> **原则**: 修改配置时，必须检查所有相关层级，使用 CLI 命令验证有效策略，而非假设单层修改即生效。
> 
> **检查方法**: 
> 1. 列出所有相关配置文件
> 2. 使用系统提供的诊断命令（如 `openclaw approvals get`）查看合并后的有效配置
> 3. 验证后再执行实际任务

### 模式 2: 跨应用 ID 隔离原则
> **适用场景**: 多应用架构下的用户身份识别（如飞书开放平台、OAuth 多应用）
> 
> **原则**: 同一用户在不同应用下的身份 ID（open_id、user_id）是隔离的，不可跨应用复用。
> 
> **最佳实践**:
> 1. 建立「应用 ID → 用户 ID」映射表
> 2. 新建任务时根据应用查表选择正确的 ID
> 3. 将映射表写入文档并定期更新

### 模式 3: 定时任务健康度监控
> **适用场景**: 任何依赖定时任务的自动化系统
> 
> **原则**: 定时任务的健康度 = 配置正确性 × 持续监控 × 快速恢复
> 
> **监控指标**:
> - lastRunStatus: 上次执行状态
> - consecutiveErrors: 连续错误次数（阈值告警 ≥ 2）
> - lastRunAt: 上次执行时间（超时告警）
> 
> **响应流程**:
> 1. consecutiveErrors ≥ 2 → 生成告警
> 2. consecutiveErrors ≥ 4 → 立即修复
> 3. 修复后创建测试任务验证
