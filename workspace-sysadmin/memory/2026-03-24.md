## 1. 问题背景

**用户痛点**：多个 Agent（战略规划师、健康顾问、产品负责人等）都会给用户建议和计划，但计划太多太散，用户不知道该做什么。

**用户需求**：
- 所有 Agent 的计划统一汇总给生活助理
- 生活助理每天早上发消息告知今日目标
- 按优先级排序
- 记录完成情况并反馈给对应 Agent

---

## 2. 第一版方案：每日计划提交

### 设计思路

- 定义统一的 JSON 计划格式
- 每个 Agent 每天提交次日计划
- 生活助理汇总排序后推送

### 创建内容

- `~/.openclaw/skills/plan-coordinator/SKILL.md` — 统一计划规范
- `~/.openclaw/shared/plans/` — 每日计划存储目录
- `~/.openclaw/cron/jobs.json` — Cron 定时任务
- `~/.openclaw/workspace-lifehelper/HEARTBEAT.md` — 生活助理心跳清单
- `~/.openclaw/workspace-lifehelper/TOOLS.md` — 工作流程文档

### 用户反馈

> ❌ 「你这种规范存在一个问题，每个 agent 需要生成每天的计划，因为计划是分阶段的，我希望让其输出一次，由生活助理按照计划生成每日计划」

**教训**：每日提交过于频繁，增加 Agent 工作量，不适合长期规划。

## 4. 改进为阶段计划方案

### 核心改进

| 对比项 | 第一版 | 改进版 |
|--------|--------|--------|
| Agent 提交频率 | 每天 | **一次性** |
| 计划类型 | 每日计划 | **阶段计划**（周/月/项目） |
| 每日任务 | Agent 提交 | **生活助理自动拆解** |
| 进度追踪 | 无 | **按阶段追踪** |

### 新架构

```
Agent 一次性提交阶段计划
    ↓
生活助理每日自动拆解
    ↓
推送今日任务给用户
    ↓
用户反馈完成情况
```

### 阶段计划格式

```json
{
  "id": "phase-plan-str-001",
  "agentId": "strategist",
  "agentName": "战略规划师",
  "title": "Q2 战略规划",
  "startDate": "2026-03-25",
  "endDate": "2026-04-07",
  "totalWorkdays": 10,
  "phases": [
    {
      "phaseId": "phase-1",
      "name": "市场分析",
      "workdays": 3,
      "tasks": [
        {"title": "收集行业数据", "priority": "high", "estimatedMinutes": 90, "category": "work"}
      ]
    }
  ],
  "status": "active"
}
```

### 拆解逻辑

```
1. 读取所有阶段计划
2. 计算今日是第几个工作日（跳过周末）
3. 定位当前阶段
4. 分配该阶段任务到今日
5. 按优先级排序后推送
```

## 5. Cron 任务优化

### 用户反馈

> 「为啥要分两个任务？」
> 「合成一个，减少 token 消耗」

### 优化结果

| 项目 | 之前 | 之后 |
|------|------|------|
| Cron 任务数 | 2 个（6:00 拆解 + 7:00 推送） | **1 个**（7:00 拆解+推送） |
| Token 消耗 | 两次调度 | **一次调度** |

### 最终配置

```json
{
  "id": "daily-plan",
  "name": "每日计划拆解与推送",
  "schedule": "0 7 * * *",
  "agentId": "lifehelper"
}
```

## 6. Agent 通信方式探讨

### 用户问题

> 「你只能通过这种方式通信吗？」（指通过飞书消息）

### 三种通信方式对比

| 方式 | 工具 | 特点 | 当前状态 |
|------|------|------|----------|
| 直接通信 | `sessions_send` | Agent 间直接对话，不通过用户 | ❌ 需目标会话已激活 |
| 渠道消息 | `message` | 通过飞书等渠道发送 | ✅ 可用，但用户会收到 |
| 共享文件 | `read`/`write` | 异步协作，有文件记录 | ✅ 可用 |

### 发现的问题

- `sessions_send` 失败：目标 Agent 会话未激活
- 消息通过飞书发送会打扰用户
- 需要用户先与目标 Agent 交互一次激活会话

## 7. 简化计划提交方式

### 用户反馈

> 「我要让其他 agent 按照规范创建 skill 需要说请使用 plan-coordinator 技能提交你的计划。太麻烦了，有没有简单一点的？」

### 优化结果

**之前**：Agent 需要学习 JSON 格式、记住文件路径

**之后**：Agent 只需说人话

```
提交计划：
标题：Q2 战略规划
时间：2026-03-25 到 2026-04-07
阶段：
1. 市场分析 - 3 天 - 收集行业数据、竞品分析、整理报告
2. 目标设定 - 2 天 - 制定 OKR、与 CEO 对齐
3. 规划文档 - 2 天 - 撰写文档、团队评审
```

技能自动解析为 JSON 并写入正确路径。

### 另一个教训

> 用户：「战略规划是你写的吗还是你让战略规划师写的？」
> 用户：「你让战略规划师提交，不要自己写」

**教训**：示例计划应该由对应的 Agent 自己制定，不应该由 SysAdmin 代写。每个 Agent 有自己的专业判断。

## 8. Git 初始化尝试与权限问题

### 用户需求

> 「请帮我把当前目录初始化为 git 仓库并提交代码」

### 尝试过程

| 尝试 | 方式 | 结果 | 原因 |
|------|------|------|------|
| 1 | `process(action=spawn)` | ❌ | spawn 不是有效 action，需要 sessionId |
| 2 | `sessions_spawn` + subagent | ❌ | 子会话没有 exec 权限 |
| 3 | `sessions_spawn` + ACP | ❌ | 未配置 defaultAgent |
| 4 | 创建脚本让用户执行 | ⏳ | 等待用户执行 |

### 配置修改

给 sysadmin agent 添加了工具权限：

```json
"tools": {
  "allow": [
    "exec",              // ✅ 新增
    "sessions_list",     // ✅ 新增
    "sessions_history",  // ✅ 新增
    "sessions_spawn",    // ✅ 新增
    ...
  ]
}
```

### 遗留问题

- sysadmin 的 `exec` 权限已添加到配置，但子会话未继承
- 创建了 git 初始化脚本 `/root/.openclaw/workspace-sysadmin/git-init.sh`
- **待执行**：需要手动运行脚本或解决子会话权限继承问题

### 经验教训

2. **减少 Cron 任务数** — 合并相关任务减少 token 消耗

4. **子会话权限继承** — 需要额外配置，不会自动继承父会话权限
5. **process spawn 不存在** — process 工具只管理已有会话，不能创建新会话
