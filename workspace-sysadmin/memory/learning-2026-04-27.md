## 2. 【想】这些问题背后的模式是什么？

### 2.1 模式一：凭证管理盲区（重复出现，影响面最广）

**表现**：
- Git Token 过期 → 连续 5 天推送失败
- 飞书 open_id 跨应用错误 → main-weekly-learning 连续 13 天失败
- 飞书 delivery 缺少 target → 多个 Cron 任务持续报错

**根因**：
- 凭证（Token、open_id）没有健康检查机制，过期后只能等任务失败才发现
- 多 Agent 架构下，每个 Agent 有独立的飞书应用和 open_id，配置分散
- 新建 Cron 任务时，delivery 配置容易遗漏 target/channel 字段

**可迁移模式**：**「凭证即基础设施」** — 凭证管理不是配置问题，而是基础设施问题。没有健康检查的凭证 = 定时炸弹。

### 2.2 模式二：配置变更的涟漪效应（间接影响）

**表现**：
- 修改 exec 审批配置后，已有 session 不自动更新权限
- 飞书应用权限开通后需等待 5-10 分钟生效
- 腾讯云防火墙规则添加后需等待 30 秒-数分钟生效

**根因**：
- OpenClaw 的多层配置体系（openclaw.json + exec-approvals.json + session 缓存）
- 外部服务（飞书、腾讯云）的配置变更有生效延迟

**可迁移模式**：**「配置变更三问」** — 改配置前问自己：(1) 有几层配置？(2) 已有 session 会受影响吗？(3) 外部服务有生效延迟吗？

### 2.3 模式三：自动化容错机制的有效性

**表现**：
- LLM 超时 fallback（qwen3.5-plus → qwen3-coder-plus/glm-5）：✅ 正常工作
- 日志轮转阈值判断（31-43MB < 50MB）：✅ 正常工作
- Cron 任务 error backoff 机制：✅ 触发正常

**根因**：
- 系统设计时考虑了容错路径（fallback、阈值判断、退避机制）
- 这些机制在上周多次验证有效

**可迁移模式**：**「容错设计是自动化的生命线」** — 没有 fallback 的自动化 = 脆弱；有 fallback 的自动化 = 韧性。

### 2.4 模式四：问题类型的持续性 vs 新增性

**表现**：
- 4-24 新增 4 类问题 → 4-25 延续 + persona.md 新问题 → 4-26-27 回归持续性
- 上周 7 天中，仅 4-24 和 4-25 有新增问题类型
- 大部分时间是「已知问题的持续」而非「新问题的爆发」

**根因**：
- 系统进入「稳定运行期」，新增问题减少
- 但已知问题（飞书凭证、Git Token）长期未修复，形成「慢性噪音」

**可迁移模式**：**「慢性噪音比急性问题更危险」** — 急性问题会触发紧急响应，慢性噪音会被习惯化而忽略。

## 3. 【变】如何优化维护流程？

### 3.1 从「事后发现」到「事前预防」

**现状**：凭证过期后等任务失败才发现（Git Token 过期 5 天后才发现）

**优化方向**：
- 建立凭证健康检查机制（REQ-003 已纳入 Sprint 3 候选）
- 在每日巡检中增加「凭证有效期检查」步骤
- 对关键凭证设置预警阈值（如到期前 7 天预警）

### 3.2 从「分散配置」到「集中管理」

**现状**：7 个 Agent 各有独立的飞书 open_id，配置分散在 MEMORY.md 的表格中

**优化方向**：
- 考虑将所有 Cron 任务的 delivery 统一走 sysadmin 渠道（单一飞书应用）
- 或在 openclaw.json 中建立「open_id 映射表」，集中管理
- 新建 Cron 任务时，从映射表自动获取正确的 open_id

### 3.3 从「被动巡检」到「主动告警」

**现状**：每日巡检发现异常后记录，但告警依赖用户查看日报

**优化方向**：
- 对 fatal 和连续 3 天以上的 error 设置即时告警
- 对「慢性噪音」问题（如 main-weekly-learning 连续 13 天失败）设置升级告警
- 周报中增加「问题趋势分析」，区分新增/持续性/已解决

### 3.4 从「单点维护」到「体系化维护」

**现状**：每次问题发生后记录到 MEMORY.md，但缺乏结构化的知识库

**优化方向**：
- 将常见问题的排查流程固化为「维护手册」（如飞书消息投递排查流程）
- 建立「维护模式库」，将可迁移的模式分类归档
- 每周学习反思不仅是记录，更是模式提取和流程优化

## 📌 可迁移的维护模式（提取 3 个）

### 模式 1：凭证即基础设施
> **核心**：凭证管理不是配置问题，而是基础设施问题。没有健康检查的凭证 = 定时炸弹。
>
> **触发场景**：Token 过期、open_id 错误、权限缺失
>
> **应对策略**：
> 1. 建立凭证清单（名称、类型、过期时间、关联任务）
> 2. 设置健康检查（每日巡检中检查凭证有效性）
> 3. 设置预警阈值（到期前 N 天预警）
>
> **适用范围**：所有涉及外部 API 调用的自动化系统

### 模式 2：慢性噪音比急性问题更危险
> **核心**：急性问题会触发紧急响应，慢性噪音会被习惯化而忽略。
>
> **触发场景**：同一问题连续出现 >3 天、warning/error 数量稳定但不为零
>
> **应对策略**：
> 1. 在日报中标记「慢性噪音」问题（连续失败天数）
> 2. 连续 >3 天的问题自动升级为 P1 优先级
> 3. 周报中统计「慢性噪音」数量，推动根治
>
> **适用范围**：所有持续运行的系统运维

### 模式 3：容错设计是自动化的生命线
> **核心**：没有 fallback 的自动化 = 脆弱；有 fallback 的自动化 = 韧性。
>
> **触发场景**：LLM 超时、网络波动、外部服务不可用
>
> **应对策略**：
> 1. 关键路径必须有 fallback 机制（如 LLM 超时 → 降级模型）
> 2. 定期验证 fallback 机制是否有效（如手动触发超时测试）
> 3. 记录 fallback 触发频率，评估是否需要优化主路径
>
> **适用范围**：所有依赖外部服务的自动化流程

## 反思总结

上周系统进入「稳定运行期」，新增问题较少（仅 4-24 和 4-25 有新增），但**慢性噪音问题**（Git Token 过期、飞书凭证缺失）持续发酵，暴露了凭证管理和慢性问题升级机制的不足。

核心收获：
1. **凭证健康检查**是当务之急（REQ-003 需尽快落地）
2. **慢性噪音**需要独立的监控和升级机制，不能与急性问题混为一谈
3. **容错机制**（LLM fallback、日志轮转阈值）经受了验证，是系统韧性的关键
