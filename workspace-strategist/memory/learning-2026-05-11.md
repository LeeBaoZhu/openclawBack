### 2. 【想】这些战略问题背后的模式是什么？

#### 模式一：结构性执行障碍 — "想推送但推不动"

**现象**：
- 每周反思都提出改进措施，其中至少 1 项是"主动询问用户 X 状态"
- 这类措施 5 周来从未执行（0/15+ 累计）
- 每次反思都归因于"被动响应""缺少触发机制"

**深层诊断**：
这不是行为问题（不够主动），而是**结构性障碍**。Strategist 运行在 `cron:strategist-weekly-learning` 的 isolated session 中：
- session 启动 → 读文件 → 写反思 → session 结束
- 没有能力在反思完成后向用户发送消息
- "询问用户家和说状态"需要跨 session 通信，但未配置此通道

**类比**：就像一个顾问每周进办公室写一份报告，然后锁门离开——报告写得再好，如果没人看、没人执行，就没有任何实际效果。

#### 模式二：反思循环的"输出崇拜"

**现象**：
- 5 周来反思文档越来越长（04-06: 6KB → 04-27: 19KB → 05-04: 19.5KB）
- 分析越来越深入（增加了"用户侧重大事件""战略视角解读""模式数量递增"）
- 但执行率从 0% 到 17% 再回到 0%，无任何正向趋势

**深层诊断**：
反思本身成了"产出物"而非"改进引擎"。每周投入大量 token 写分析、提取模式，但改进链路在第一步（执行）就断裂了。这是一种**认知工作的自我欺骗**——把"更深的分析"当成"真正的进步"。

对应到用户的信息设计哲学（好报告 9 条标准），反思文档违反了"可执行优于可描述"原则。

#### 模式三：战略文档"僵尸化"

**现象**：
- current.md 描述的用户状态是 03-17 的（55 天前）
- okr.md 所有进度仍为 0%，像一个从未启动的计划
- 用户实际在做的（AI Agent 建设、家庭健康系统化）并不在战略文档中

**深层诊断**：
战略文档已从"活文档"变成"僵尸文档"——形式存在，实际上已死亡。不是因为内容错误，而是因为**无人维护→与现实脱节→更无人维护**的正反馈循环。

这与用户的"故障即进化"模式形成鲜明对比：用户遇到 Git Token 过期 → 2 天修复 → 沉淀经验；战略文档漂移 55 天 → 无任何修复动作。

### 3. 【变】如何优化规划方法？

#### 变革 1：打破结构性障碍 — 建立 Strategist 主动推送通道

**现状**：Strategist 只能"响应"不能"发起"对话
**方案 A**：在 weekly learning cron 中增加一个步骤——反思完成后，通过 `sessions_send` 向用户发送总结消息
**方案 B**：创建独立的"Strategist 周报推送"cron 任务，在每周一 10:30 向用户推送周反思摘要
**方案 C**：在反思文件写入后，通过 systemEvent 将摘要推送到用户聊天中

**推荐**：方案 A（最轻量）——在现有 cron 任务中追加一个 sessions_send 调用。但需要确认：cron isolated session 是否能调用 sessions_send 跨 session 通信。

#### 变革 2：打破反思死循环 — 引入"电路断路器"

**规则**：如果同一类改进措施连续 3 周执行率为 0%，则：
1. **禁止**再提出同类改进措施（停止产生无效输出）
2. **强制升级**：将该障碍标记为"结构性障碍"而非"行为问题"
3. **停止常规反思**：直到结构性障碍被解决，每周反思只写一行：`结构性障碍未解除，本周跳过深度反思`

**目的**：防止反思本身变成认知负债（消耗 token 制造无用的分析）。

#### 变革 3：战略文档从"定期更新"改为"事件驱动更新"

**现状**：期望定期维护文档（每周/每月），但无人执行
**改进**：
- 仅在以下时机更新战略文档：
  1. 用户主动发起战略对话后
  2. 重大外部事件（如用户做出关键决策）
  3. 季度复盘时
- 放弃"每周更新"的期望（这个期望本身已证明不现实）
- 在文档顶部注明"最后实质性更新"vs"最后格式更新"

#### 变革 4：OKR 从"进度跟踪"改为"存在性检测"

**现状**：OKR 13 个 KR 全为 0%，数字没有信息量
**改进**：
- 将 OKR 的"进度"字段改为"最后活动时间"
- 如果一个 KR 超过 30 天无活动，标记为 🧟（僵尸 KR）
- 目标是让用户看到哪些目标实际上已被放弃，触发显式决策（重新激活 or 正式放弃）

## 可迁移的战略模式（提取 3 个）

### 模式 1：结构性障碍 vs 行为障碍的区分诊断

**核心洞察**：当改进措施连续 3 周以上执行率为 0% 时，问题大概率是**结构性的**（工具限制、权限缺失、流程断裂），而非**行为性的**（不够主动、忘记执行）。

**诊断框架**：
- 问：这个改进措施需要什么能力？
- 问：当前系统是否提供这个能力？
- 如果否 → 结构性障碍，应先解决结构问题再谈改进
- 如果是 → 行为障碍，可以谈触发机制、习惯养成

**应用场景**：任何持续失败的改进循环、反思→不执行→再反思的死循环

### 模式 2：反思产出的收益递减检测

**核心洞察**：反思本身有边际收益递减。当连续多周反思输出相同洞察、提出相同改进建议时，继续增加分析深度不会改变结果——需要"电路断路器"强制降级。

**检测标准**：
- 同一类改进建议出现 ≥3 次且执行率 = 0% → 停止反思该类问题
- 反思文档长度持续增长但执行率持续下降 → 反思已变成"输出崇拜"
- 上周改进措施执行率 = 0% → 本周反思必须降级（减少 token 投入）

**应用场景**：任何定期反思机制（周报、复盘、OKR 回顾），防止反思本身成为认知负债

### 模式 3：战略文档的生命周期管理

**核心洞察**：战略文档不是"写了就永远有效"的文物，它有生命周期：活跃期 → 维护期 → 漂移期 → 僵尸期。关键是在进入漂移期时就做出显式决策（续命 or 归档），而不是让它缓慢死亡。

**生命周期**：
```
活跃期（0-14 天）→ 维护期（15-30 天）→ 漂移期（30-60 天）→ 僵尸期（>60 天）
                                                          ↑
                                                    应在此处决策
```

**应用场景**：任何需要定期维护的战略文档、OKR、路线图。设定"最后活动时间"字段，超过 30 天自动标记，超过 60 天触发归档/刷新决策。
